9 Failure path — B is offline when A sends

Server's Hub lookup misses. Server returns an error frame to A. A's outbox keeps the message at status='sent' and never promotes it. There is no FCM, no server queue — the bytes are gone after the failed write attempt.

10 Outbox retry — both online again

A's RelayClient.onReconnected fires (or a periodic 60s timer). MessageService queries the outbox for any row with status IN ('pending','sent') AND last_try < now()-10s and re-sends each. Recipient dedupes by msgId; if already persisted, it re-emits the ack.

This is the entire reliability mechanism. No server queue, no FCM. The sender's device is the source of truth for "did this message make it."

What lives on the server, total

  • RAM, per active connection: the pubkey of who's connected + the open socket. Gone on disconnect or VM reboot.
  • Disk: the Go binary, the systemd/Docker unit, optional TLS cert, container logs (configurable retention).
  • Database: none. No phonebook. No FCM tokens. No message queue. No user table.

What the server can see while online

  • Which pubkeys are connected right now (must, for routing)
  • Pairs: who is sending to whom, when, how many bytes
  • Cannot see: message body, msgId, message type (text/ack/invite/etc.), display names, group structure

Comparison vs. today

  • Today: server holds pubkey → FCM token in Firestore, has Firebase Admin credentials, logs deliver_offline + wake events.
  • Version B: server holds nothing on disk beyond logs you choose to keep. Runs on any $5/month VPS. No Firebase project required.
