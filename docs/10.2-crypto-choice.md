# Phase 10.2 — Crypto Choice

**Date:** 2026-05-19
**Decision:** Adopt `libsignal_protocol_dart` (MixinNetwork) for end-to-end encryption in Phase 10.2 and beyond.

## Evaluation summary

| Property | libsignal_protocol_dart |
|---|---|
| Version evaluated | 0.8.0 (published 2026-04-28) |
| Repository | https://github.com/MixinNetwork/libsignal_protocol_dart |
| Repo last push | 2026-04-28 (active) |
| Stars / weekly downloads / pub.dev points | 172 / 5.07k / 150 |
| Production users | Mixin Messenger and other Signal-protocol Dart consumers |
| Status | Not archived, not deprecated, recent dependency upgrade in the latest release |
| Provides | Full Signal Protocol: X3DH initial key agreement + Double Ratchet message keys, with the standard SessionBuilder / SessionCipher / PreKeyBundle surface |
| **License** | **GPL-3.0** |

## Why Path A

- The Signal Protocol's forward secrecy (Double Ratchet) and post-compromise security (re-randomization on each round trip) are real, meaningful properties that we'd be giving up in the Path B fallback.
- The binding is actively maintained — fresh version published 21 days before this decision.
- A community-audited, widely-used Dart implementation is materially less risky than a hand-rolled X25519 + ChaCha20-Poly1305 wrapper, even when the underlying primitives are sound.
- The Heartbeat product trajectory (start small with Tatha; explore growing into a secure P2P chat product) lines up with the security posture of Signal / SimpleX / Briar — all GPL-3.0.

## What Path A costs

- **License entanglement:** heart-beat-v3 must be GPL-3.0 (or a copyleft-compatible license) when distributed. This forecloses a closed-source commercial path. The Tatha use case is unaffected; future distribution must offer source under GPL-3.0. The same posture as Signal, SimpleX, Briar.
- **Bundle size:** libsignal_protocol_dart pulls in a sizeable Dart-only crypto layer. Acceptable; the v3 APK is small to begin with.
- **Maintenance dependency:** if the MixinNetwork project goes dormant, we'd need to migrate (likely to FFI bindings for the official Rust libsignal). Mitigated by the `CryptoService` interface layer (Task 7) that keeps the rest of the app binding-agnostic.

## Path B (rejected)

X25519 + ChaCha20-Poly1305 via the `cryptography` package (already in pubspec). Apache-2.0, no license entanglement. Rejected because the forward-secrecy gap is a material security regression for an app whose explicit goal is end-to-end privacy. Documented in the Phase 10.2 plan's Appendix A for future reference if libsignal_protocol_dart later becomes unmaintained.

## When to revisit

- If the MixinNetwork repo goes dormant (no commits / no release for 18+ months) — switch to FFI bindings for the official Rust libsignal, or accept the Path B trade-off explicitly.
- If a closed-source / commercial product becomes the goal — switch off libsignal_protocol_dart at that point and re-evaluate the security model.
